# PhantomNet Agent Usage Instructions

This document provides a quick guide on how to start and interact with the PhantomNet Agent, detailing its exposed APIs and backend communication mechanisms.

## How to Start the Agent

The PhantomNet Agent is designed to run in a Python 3 environment. It can be started using the following command from the project's root directory:

```bash
python -m phantomnet_agent.main --config config/agent.yml --mode collector-only
```

*   `python -m phantomnet_agent.main`: Executes the `main.py` module as part of the `phantomnet_agent` package.
*   `--config config/agent.yml`: Specifies the agent's configuration file. Ensure `config/agent.yml` is correctly set up for your environment.
*   `--mode collector-only`: Starts the agent in a mode where it primarily runs collectors and does not require a fully functional backend manager for initial boot. This also enables the "SAFE AI MODE" (configured in `agent.yml`) which uses simulated AI reasoning due to Termux constraints.

### Important Note for Termux (Android, aarch64) Users:

Due to Termux's environment limitations, running long-lasting Python processes in the background with standard `&` or `nohup` might result in premature termination. For persistent execution, it is highly recommended to use a terminal multiplexer like `tmux` or `screen`:

1.  **Install `tmux` or `screen`** (if not already installed):
    ```bash
    pkg install tmux  # or pkg install screen
    ```
2.  **Start a new session:**
    ```bash
    tmux new -s phantomnet-agent  # or screen
    ```
3.  **Run the agent within that session:**
    ```bash
    python -m phantomnet_agent.main --config config/agent.yml --mode collector-only
    ```
4.  **Detach from the session** (the agent will continue running):
    *   For `tmux`: Press `Ctrl+b` then `d`
    *   For `screen`: Press `Ctrl+a` then `d`
5.  **To reattach and check status:**
    ```bash
    tmux attach -t phantomnet-agent  # or screen -r
    ```

## Exposed API Endpoints (Local)

The agent exposes a local FastAPI instance (by default on `http://127.0.0.1:8000`) for control and monitoring. These endpoints are designed for local interaction and are not exposed to the public internet by default.

*   **Health Checks:**
    *   `GET /api/agent/health`: Returns a basic snapshot of the agent's health and operational status.
    *   `GET /api/agent/status`: Returns a more detailed status of the agent, including overall status and a summary of component health.
    *   `GET /api/agent/components`: Provides detailed health status for each individual agent component (collectors, bus, etc.).

*   **Command Control:**
    *   `POST /api/agent/command`: Accepts JSON commands to be processed by the agent's orchestrator (e.g., "run_scan", "control_collector").
    *   `GET /api/agent/commands/status/{command_id}`: Retrieves the current status and results of a previously issued command.

*   **Log Streaming:**
    *   `GET /api/agent/logs/tail`: Returns the most recent `N` structured log entries from the agent's `logs/agent_output.log` file in JSON format for debugging.
    *   `WS /api/ws/logs`: A WebSocket endpoint for real-time streaming of structured log events to a connected UI or console.

*   **Collector Control (Specific):**
    *   `POST /api/control/collector/{collector_name}/start`: Starts a specific collector.
    *   `POST /api/control/collector/{collector_name}/stop`: Stops a specific collector.
    *   `POST /api/control/plugins/reload`: (Conceptual) Reloads all plugins.

## How the Agent Connects to the Backend

The PhantomNet Agent interacts with a separate backend API service for various functionalities:

*   **Registration:** At startup, the agent generates its cryptographic keys and attempts to register itself with the backend manager (configured via `agent.manager_url` in `agent.yml`). This involves sending its public key and, if successful, receiving a signed certificate.
*   **Authentication:** The agent uses JSON Web Tokens (JWTs) for authenticating its communication with the backend. After registration, it uses its private key to sign JWTs, which are included in the `Authorization` header of outgoing HTTP requests (e.g., heartbeats, event forwarding).
*   **Heartbeats:** The `AgentHealthMonitor` periodically sends heartbeats (via HTTP POST) to the backend's `/agents/heartbeat` endpoint to report its operational status and component health. These heartbeats are authenticated with JWTs.
*   **Log & Event Forwarding:** The `LogForwarder` component captures all structured log events generated by the agent and continuously forwards them in batches (via HTTP POST) to the backend's `/api/v1/logs/ingest` endpoint. Other critical events (e.g., AI analysis results) can also be forwarded via dedicated endpoints (e.g., `/api/v1/analysis/ingest`) using the configured HTTP transport.
*   **mTLS Support (Optional):** The agent supports mutual TLS (mTLS) for enhanced security in communication with the backend, if enabled and configured with appropriate certificate paths in `agent.yml`.
